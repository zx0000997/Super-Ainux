# Super Ainux System (SAS) — 系统开发规格说明书

**版本**: 1.0  
**受众**: 系统架构师、后端/基础设施工程师、AI 工程、安全与合规、前端（看板）  
**目标**: 读者读完即可拆分里程碑、接口、数据模型与实现任务。

---

## 1. 文档范围与术语

### 1.1 本文档定义什么

- SAS 的产品边界、子系统划分、**任务优先**的运行模型。  
- **双 LLM 治理**（Operator / Supervisor）的职责、权限委托与先审后行流程。  
- 本地数据面（文档入库、检索、模板）、**统一审计日志**、任务与数据的绑定模型。  
- 阻塞类事件（凭证、付费/额度、选型）、通知渠道、交付物规范。  
- 与 **Linux 发行版/用户态**相关的实现要求（不重复通用内核编译教程；内核基线见仓库 `OS_DEVELOPMENT_GUIDE.md`）。

### 1.2 本文档不替代什么

- 具体编程语言与框架选型（可在实现阶段由技术栈 ADR 固定）。  
- 第三方模型厂商 API 细节（需随供应商文档演进）。

### 1.3 术语表

| 术语 | 含义 |
|------|------|
| **SAS** | Super Ainux System：基于 Linux 的、以 AI 为操作主体、人类提出需求与方案选择的系统。 |
| **Operator** | 操作员 AI：负责产制 Plan、执行任务；**不得**拥有任意删除权；不直接写入审计日志。 |
| **Supervisor** | 管理员 AI：审查/修改 Plan、授权临时能力、参与完成度核验；与 Operator **必须异源模型**（不同供应商或不同模型族）。 |
| **任务 (Task)** | 系统的一等处理对象；执行与检索默认带 `task_id` 作用域。 |
| **审定计划包 (Approved Plan Package)** | 经 Supervisor 批准后、带哈希不可变、唯一允许执行的计划工件。 |
| **统一审计 (Audit Log)** | 仅追加的事件总线：监管授权、任务状态迁移、删除提案、交付物等。 |
| **信息整理** | 入站文件抽取、入库、索引、定期整理与去重提案。 |
| **用户画像** | 多维度偏好与属性，用于增强 LLM 输入；可短时授权共享。 |
| **任务看板** | 稳定的人类控制面：观测任务状态、通知、结果入口（**非**强制审批闸门）。 |

## 2. 愿景与目标

### 2.1 产品陈述

SAS 运行在 **x86_64** 环境（典型：**家用迷你服务器**），以 **Linux 内核**为底座。系统的**操作主体是 AI**（Operator），**监管主体是另一异源 AI**（Supervisor）。人类通过需求输入与**方案选择**参与，但不作为日常“点击批准”的角色。

### 2.2 成功标准（工程可验证）

- **先审后行**：任何影响生产态的变更必须来自 **Approved Plan Package**。  
- **可审计**：监管授权、状态迁移、删除提案、交付物均可从 **同一条 Audit Log** 追溯。  
- **任务绑定**：业务写入与默认检索以 **`task_id` 为主键作用域**（见第 7 节例外）。  
- **本地可恢复**：失败时可通过快照/基线/回滚路径恢复服务（不强制离线可用；允许依赖在线模型与在线仓库）。  
- **数据调用顺序**（任务执行时）：**临时输入 > 本地按时间由近及远 > 互联网公开数据**。

---

## 3. 架构原则

1. **机制优于建议**：监管是系统服务与策略，不是“另一个聊天角色”。  
2. **异源双脑**：Operator 与 Supervisor **不得同源模型**（配置层强制）。  
3. **Operator 无删除**：Operator 不具备对业务数据的删除权限；删除走**监管审核 + 受信作业**。  
4. **任务优于文件**：人类契约面是任务；文件是执行层与交付层产物。  
5. **单一事实审计**：全局 **Audit Log** 覆盖授权、任务机、删除、交付。  
6. **入库再检索**：入站文档进入本地库后再供 AI 检索；**交付时**才生成对外新文件（及消息/邮件等）。

## 4. 系统上下文与外部实体

```
[人类] -- 需求/方案选择 --> [任务看板 / 移动端]
                              |
[AI Operator] <-- 受控执行 --> [SAS 核心服务]
[AI Supervisor] <-- 审查/授权 --> [SAS 核心服务]

[SAS] <--> [本地用户数据库：文本/媒体/文件存储]
[SAS] <--> [在线仓库：工具 / 文档模板 / Skills]
[SAS] <--> [互联网公开数据]
[SAS] --> [输出：文档 / 邮件 / 媒体 / 广播等]
```

### 4.1 部署假设

- **CPU**: x86_64。  
- **网络**: 允许访问公网模型 API 与在线仓库（**不**要求离线可用）。  
- **隐私**: 允许数据出网至模型厂商（以用户策略与合规配置为准）。

## 5. 逻辑架构 — 子系统

### 5.1 监管子系统 (Supervision)

**职责**

- 双 AI 协同：**先审后行**；维护临时授权与撤销。  
- 对 **数据库增删改查** 做策略控制：在 Plan 通过后，由 Supervisor 侧逻辑向 Operator 颁发**限时、可撤销**的授权令牌（或等价机制）。  
- **授权模型（既定）**  
  - 通过后：Operator 可获得 **增加**、**查询**（建议实现为 **task 范围优先**，避免“全库任意读”）；以及对**相关数据**的 **修改** 权限。  
  - Operator **没有删除**权限。  
- **删除与去重**：信息整理子系统产生**删除/逻辑删除提案** → 监管审核 → 由**非 Operator** 的受信执行者执行 → 记 Audit。

**必须产出**

- 授权 API：`grant` / `revoke` / `renew`（含过期时间与作用域）。  
- 策略拒绝与修改 Plan 的结构化原因码（便于任务机与看板）。

### 5.2 任务子系统 (Task)

**职责**

- 任务生命周期与状态机。  
- 收集用户意图、关联用户画像维度 → 提交 Operator 生成 **Plan** → 提交 Supervisor **审查/修改** → 生成 **Approved Plan Package** → **仅执行审定包**。  
- 执行阶段：按审定包拉取 **工具、文档模板、Skills**（在线仓库）；若涉及付费 → 走 **额度/策略**（见 9 节）；若多模板 → **选型策略**（见 9.3）。  
- 按既定顺序使用：**临时输入 > 本地由近及远 > 互联网**。  
- 异常：执行中问题可升级由 Supervisor 介入（重规划/约束/中止策略需可配置）。  
- 完成：Supervisor **核验是否真实完成**；未达标则退回 Operator 继续执行（需最大迭代/超时策略防止死循环）。  
- **交付**：以消息、邮件、文件等形式输出；交付动作记 Audit。

**任务状态（参考）**

`New` → `Planning` → `PendingReview` → `Approved` → `Executing` → `Verifying` → `Completed` → `Delivered`  
（失败/回滚/阻塞类状态见第 8 节）

**必须产出**

- 任务 API：创建、查询、状态迁移、绑定 `task_id`。  
- Plan / 审定包版本化与哈希校验。  
- 执行器：只接受审定包哈希对应的指令集（immutable artifact）。

### 5.3 信息整理子系统 (Information Organization)

**职责**

- 入站：**文档（PDF/Office 为主）**、文字、多媒体等 → 抽取与归档。  
- 文档长文本入 **`sa_file`**（或等价表）；提炼索引以支持检索与对话引用。  
- **版式/模板** 元数据入 **`sa_template`**（命名建议统一 *template*，避免 *temple* 歧义）。  
- 多媒体按时间落本地存储，数据库保存关联与元数据。  
- **定期整理**（如每日凌晨，可配置）：去重检测到重复/冗余 → **生成删除提案** → **监管审核** → **逻辑删除**（非物理删除优先）。

**检索策略（既定）**

- 文档主场景：**混合检索**（关键词/BM25/全文 + 可选向量）+ 元数据过滤；检索返回需带 **引用锚点**（页码/块 ID）。  
- 向模型侧默认只投喂 **top-K 相关块**，避免整库塞上下文。

### 5.4 分析子系统 (Analysis)

**职责**

- 基于任务反馈与行为逐步更新 **多维度用户画像**，作为 LLM 输入，替代非结构化的“长期记忆”堆叠。  
- **画像维度（包含但不限于）**：基本信息；视觉偏好（色彩、布局、形状）；表达偏好等。  
- **短时共享**：画像片段可授权给其他用户；需 **范围、时长、撤销与审计**。

### 5.5 高级日志 / 统一审计 (Advanced Log = Audit)

**职责**

- **唯一追加** 的审计事件总线，至少包含：  
  - 监管：**授权/回收/策略变更/否决**  
  - 任务：**状态迁移**（注明触发者：system / operator_agent / supervisor_agent / policy）  
  - 数据治理：**删除提案、审核结论、逻辑删除执行**  
  - 交付：**交付物生成、外发记录（邮件/消息/文件路径或对象键）、校验哈希**

**约束**

- Operator 进程 **不直接写** Audit；由受信服务落账。  
- 与 **通知投递** 可关联同一 `event_id` 以便幂等与追踪。

### 5.6 支撑模块

| 模块 | 职责 |
|------|------|
| **任务看板 (Web/桌面)** | 任务列表、状态、阻塞原因、通知、交付入口；可选方案对比（人类选方案，但不强制每步点击）。 |
| **移动端** | 与看板一致信息架构；推送通知。 |
| **AI 专用浏览器** | 受控网页访问能力；需与监管授权、Audit 对齐。 |
| **邮件服务** | 系统邮件发送；可与通知子系统共用 SMTP。 |
| **通知子系统** | 看板内通知 + APP/邮件推送；缺凭证/付费/选型等事件。 |

## 6. 数据模型要点

### 6.1 `task_id` 贯穿

- **默认**：任务产生的写入与变更必须带 `task_id`。  
- **检索默认过滤**：按 `task_id` 可见集合 + 策略允许的共享/全局归档区。  
- **例外**：纯入站归档尚未关联任务时允许 `task_id IS NULL`，但必须存在 **`ingest_batch_id` / `source_id`**；后续任务通过 **link** 操作关联并记 Audit。

### 6.2 核心表（逻辑名，实现可调整）

- `tasks`：状态、优先级、验收条件引用、审定包哈希。  
- `sa_file`：文档抽取文本、分块、索引、版本。  
- `sa_template`：模板/版式元数据。  
- `media_objects`：路径、哈希、时间、关联。  
- `user_profile`：多维度画像与版本。  
- `audit_events`：追加-only，结构化 JSON payload + 哈希链（推荐）。  
- `quota_ledger`：额度账户、预占、扣费、人工充值记录（MVP）。

### 6.3 审定计划包 (Approved Plan Package)

必须包含：

- `plan_id`、`task_id`、**内容哈希**、Supervisor 批准标记、过期策略。  
- **动作序列**（可序列化）：包管理、服务、容器、文件生成、外部 API 调用等。  
- **验收标准**：可机器校验的条目（探针、查询、文件哈希等）。  
- **回滚提示**：快照 ID 或基线 ID（与本地恢复策略一致）。

---

## 7. 工作流 — 先审后行

1. Operator 产出 **Plan**（非最终可执行）。  
2. Supervisor **审查/修改**，输出 **Approved Plan Package**。  
3. 监管子系统 **颁发授权**（限时、scoped）。  
4. 执行器 **仅执行** 审定包；执行中不得改意图（变更需新包）。  
5. Verifying：收集 **证据包**（日志片段、命令结果、产物哈希、引用块）。  
6. Supervisor **完成度核验**；通过后 **交付** 并记 Audit。

---

## 8. 阻塞类事件与策略

### 8.1 付费 / 额度（MVP）

- **MVP**：**额度记账 + 人工充值**（非真实支付网关）。  
- **策略**：用户可配置 **小额免密**（后续接支付再接回）；或 **仅免费资源**（Plan 阶段硬拒绝付费依赖）。  
- **执行前预占额度**；失败进入 `BlockedPayment` 或触发 Plan 改写。

### 8.2 选型（模板/多方案）

- 可配置：**画像自动选择** 或 **提示用户选择**（看板/推送）。  
- **超时默认**：**自动选择**；建议固定 tie-break：画像匹配 > 成本最低（免费优先）> 风险最低 > 最近使用。

### 8.3 缺凭证

- 状态：`BlockedCredential`；**通知**用户（看板 + APP/邮件）。  
- Audit 记录 `credential_type` 与重试策略。

---

## 9. 通知与交付

- **渠道**：看板内 **canonical**；外发 **APP / 邮件**（幂等、可追踪）。  
- **交付形态**：消息、邮件、文件、广播等；**对外新文件主要在交付阶段生成**。  
- 交付物必须可关联 `task_id`、审定包哈希、产物哈希。

---

## 10. 安全与威胁建模（最小集）

- **异源模型**降低单供应商单密钥击穿双脑的风险；密钥分仓。  
- **最小权限**：查询/修改授权默认 **task-scoped**。  
- **提示注入**：区分不可信用户内容与系统指令；工具调用走审定包。  
- **家庭服务器暴露面**：防火墙、仅必要端口、管理面分离（实现期细化）。

---

## 11. 可观测性与运维

- **指标**：任务耗时、阻塞率、检索延迟、模型 API 错误率、额度消耗。  
- **日志**：业务日志与 **Audit** 分离；Audit 防篡改（权限 + 哈希链/只追加存储）。  
- **备份**：用户库与 Audit 的备份与保留策略。

## 12. 实施路线图（建议）

| 阶段 | 交付物 |
|------|--------|
| **M0** | 基线 Linux 镜像、单节点部署、密钥与配置管理 |
| **M1** | 任务表 + 状态机 API；Audit 骨架；空壳 Operator/Supervisor 适配层 |
| **M2** | 信息整理：PDF/Office 抽取、`sa_file` 分块与混合检索 |
| **M3** | 审定包 + 执行器沙箱；监管授权令牌 |
| **M4** | 看板与通知；额度 MVP；阻塞态联调 |
| **M5** | 交付管道（邮件/文件）；完成度自动核验 |
| **M6** | 画像与共享；去重删除提案全链路 |
