## 2. 产品愿景（已对齐）

- 基于 **Linux 内核** 的系统，部署在 **x86_64 家用迷你服务器**。  
- **操作主体是 AI**，不是传统「以人为终端用户」的桌面 OS。  
- **Operator AI**：具备 root/等价能力，可执行 shell、改配置、装包、起服务等（实现上仍建议通过受控执行器与审定包落地）。  
- **Supervisor AI**：监管；与 Operator **必须异源**（不同模型族/不同供应商），不是同一模型的两个角色。  
- **人类**：提出需求、在**多套方案间做选择**；**不要求**每一步人工点击审批。  
- **人类控制面**：稳定的 **任务看板**（观测、通知、交付入口），不是强制闸门。  
- **离线**：不强求离线可用；可依赖在线模型 API 与在线仓库。  
- **合规/隐私**：用户允许的前提下数据可出网至模型厂商。

## 3. 核心机制（已对齐）

### 3.1 先审后行（监管是机制）

- Operator 制定 **Plan** → Supervisor **审核/修改** → 生成 **审定计划包（Approved Plan Package，不可变、可哈希校验）** → Operator **仅执行审定包**。  
- **监管子系统**对数据库的增删改查做 **临时授权**；Plan 通过后由 Supervisor 侧逻辑授权 Operator：  
  - 可增加、查询（建议实现为 **按 task 范围**，避免任意全库读）；可修改相关数据；  
  - **Operator 无删除权限**。  
- 信息整理子系统中的 **重复/冗余数据逻辑删除**：先 **删除提案** → 监管审核 → 由 **非 Operator** 的受信作业执行。

### 3.2 任务优先，不是文件优先

- 系统的一等对象是 **任务（Task）**；执行与检索默认带 **`task_id`**。  
- 入站文件经抽取进入 **本地数据库**，AI 从库中检索与调用；**对外新文件主要在任务交付阶段**生成，供其它系统使用。  
- 无任务先归档时允许 `task_id` 为空，但需 **`ingest_batch_id` / `source_id`**，后续任务可 **link** 并记审计。

### 3.3 数据与检索

- 入站以 **PDF/Office 文档** 为主。  
- 调用数据方式：**高效优先** → 落地为 **混合检索**（全文/BM25 + 可选向量）+ 元数据过滤；默认 **top-K 块 + 引用锚点**，不整库塞上下文。

### 3.4 统一审计（Advanced Log = Audit）

以下共用 **一条 append-only 的 audit log**（事件总线概念）：监管授权、任务状态迁移、删除提案、交付物。  
- **Operator 进程不直接写审计**；由受信服务落账。

### 3.5 用户画像（分析子系统）

- 多维度画像（基本、视觉、表达等）作为 LLM 输入，替代非结构化的长期记忆堆叠。  
- 可 **短时授权共享** 给他人；需范围、时长、撤销与审计（规格中已要求）。

### 3.6 任务执行时的数据优先级

**临时输入 > 本地按日期由近及远 > 互联网公开数据。**

### 3.7 阻塞类事件与策略（已对齐）

| 场景 | 策略 |
|------|------|
| 付费 | MVP：**额度记账 + 人工充值**（非真实支付网关）；可配置仅用免费资源等。 |
| 选型（模板/多方案） | 可配置 **按画像自动选择** 或 **提示用户**；**超时默认自动选**（建议 tie-break：画像匹配 > 成本/免费优先 > 风险 > 最近使用）。 |
| 缺凭证 | **通知**用户；看板内通知 + **APP 或邮件推送**。 |

### 3.8 思维导图结构（用户已提供）

中心为 SAS；外围含：监管、任务、信息整理、分析、高级日志、基础模块（如 AI 浏览器、邮件服务器）；任务节点含 Plan、工具/模板/Skills（在线仓库）、公网数据、用户数据（画像 + 临时输入 + 本地库）；输出含文档/邮件/媒体/广播等。
